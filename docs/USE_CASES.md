# 🧭 Сценарии использования и граничные случаи

Документ описывает ожидаемое поведение функций, апплетов и составителя расписания. Он дополняет [USER_GUIDE.md](USER_GUIDE.md) и [SCHEDULE_WORKFLOW.md](SCHEDULE_WORKFLOW.md).

## 1. Запуск панели

### 1.1. Обычный браузер

**Дано:** открыт `https://localhost:3000/src/ui/taskpane.html` вне Excel.

**Ожидается:**

- отображается статус «Браузерный preview»;
- вкладки и поиск заполнены;
- работает стенд функций;
- можно копировать примеры формул;
- кнопки изменения книги выводят понятное сообщение «доступно только внутри Excel»;
- пустых секций нет.

### 1.2. Excel Microsoft 365

**Дано:** панель открыта через `manifest.xml`.

**Ожидается:**

- статус меняется на «Excel подключён»;
- активируются команды книги;
- формула вставляется в выбранную ячейку;
- кнопка «Проверить книгу» создаёт или восстанавливает служебную структуру.

### 1.3. Excel 2016 task pane

**Дано:** загружен `manifest-office2016.xml`.

**Ожидается:** используется облегчённая ES5-панель. Custom Functions и Shared Runtime не объявляются. Для полного legacy-функционала используется XLAM.

## 2. Генерация функций

### 2.1. Чистый checkout без `src/functions.json`

**Действие:** выполнить `npm run build`, `npm run build:dev` или `npm start`.

**Ожидается:** генератор автоматически создаёт `src/functions.json`, `docs/FUNCTIONS.md` и `docs/APPLETS.md`, после чего копирует метаданные в `dist/functions.json`.

### 2.2. Изменён каталог функций

**Действие:** добавить функцию в `function-catalog.js` и сохранить при работающем `npm start`.

**Ожидается:** watch запускает пересборку; browser preview перезагружается; новые метаданные появляются в `dist/functions.json`.

### 2.3. Неверные метаданные

**Действие:** выполнить `npm run check`.

**Ожидается:** validation или smoke-тест завершается ошибкой, если число функций не равно 114, отсутствует metadata-файл или манифест ссылается не на тот URL.

## 3. ФИО и кадровые данные

### 3.1. Полное ФИО

```excel
=PROFI.ФИОКРАТКО("Иванов Иван Иванович")
```

Ожидаемый результат: `Иванов И. И.`.

### 3.2. Лишние пробелы и регистр

```excel
=PROFI.ФИОНОРМА("  иВАНОВ   иВАН  иВАНОВИЧ ")
```

Ожидается нормализованный регистр и одинарные пробелы.

### 3.3. Неполное ФИО

Для строки `Иванов Иван` функция не должна создавать вымышленное отчество. Возвращаются только имеющиеся части.

### 3.4. Одинаковые фамилии

В расписании фамилия без достаточных инициалов считается неоднозначной. Пользователь добавляет явный вариант в `tblTeacherAliases`.

### 3.5. Телефоны разных форматов

`8 (999) 123-45-67`, `+7 999 1234567` и `89991234567` должны приводиться к единому представлению, если число цифр корректно. Необычная длина не должна молча превращаться в другой номер.

### 3.6. Пустые обязательные поля

Функции полноты и апплет аудита должны выделять отсутствующие значения, но не подставлять персональные данные автоматически.

## 4. Дубли и очистка

### 4.1. Полные дубли

Апплет подсвечивает повторные значения в выделении, не удаляя исходные строки без отдельного подтверждения.

### 4.2. Похожие, но не идентичные записи

`ООО Ромашка` и `Ромашка, ООО` обрабатываются нечётким сопоставлением, а не как гарантированный полный дубль.

### 4.3. Смешанный алфавит

Строка с латинской `A` внутри кириллического названия должна отмечаться как потенциальная ошибка.

### 4.4. Формулы в очищаемом диапазоне

Массовая операция не должна заменять формулы значениями без явного режима пользователя.

## 5. Планирование задач

### 5.1. Чистая книга

Апплет «Недельный планировщик» создаёт таблицу с датой, задачей, исполнителем, приоритетом, статусом, сроком, прогрессом и комментарием.

### 5.2. Просроченная задача

Если срок прошёл и статус не является выполненным, `СТАТУСЗАДАЧИ` возвращает `Просрочена`.

### 5.3. Выполненная задача с прошедшим сроком

Явный статус «Выполнена» имеет приоритет над расчётом просрочки.

### 5.4. Перегрузка

Функция перегруза возвращает только положительное превышение нормы и не создаёт отрицательную «перегрузку».

## 6. Сводное расписание: первый запуск

### 6.1. Полностью чистая книга

**Действие:** нажать «Составить сводное расписание».

**Ожидается:**

1. автоматически создаются все `_PROFI_*` листы и таблицы;
2. пользователь получает предложение подключить текущий групповой лист;
3. существующие пользовательские листы не удаляются и не переименовываются;
4. служебные листы после операции могут быть скрыты.

### 6.2. Часть служебных листов удалена

Повторный запуск восстанавливает только отсутствующие листы, таблицы и столбцы. Существующие строки справочников сохраняются.

### 6.3. В книге уже есть лист с похожим именем

Системные имена проверяются точно. Пользовательский лист не должен быть принят за `_PROFI_DATA` только из-за похожего заголовка.

## 7. Подключение группового расписания

### 7.1. Группа введена вручную

Пользователь выбирает режим `manual` и вводит `522`. Значение сохраняется в `tblSources`.

### 7.2. Группа из имени листа

Режим `sheet` повторно читает имя листа при обновлении источника.

### 7.3. Группа из ячейки

Режим `cell` сохраняет адрес, например `B2`. При следующем обновлении группа читается из этой ячейки заново.

### 7.4. Ячейка группы пуста

Источник получает диагностическое сообщение. Парсер не должен создавать занятия с пустой группой без предупреждения.

### 7.5. Лист был переименован

Пользователь обновляет имя листа в `tblSources` либо повторно подключает источник. Система должна явно сообщить, что прежний лист не найден.

### 7.6. Несколько групп

Каждый лист имеет собственный `SourceID` и профиль. Итоговая таблица объединяет занятия всех активных источников.

## 8. Автоопределение парсера

### 8.1. Стандартный шаблон

Система находит строку `уч. недели`, первую неделю, строку месяцев, сетку, блок `Обозн`, столбцы лекторов и других преподавателей.

### 8.2. Маркер написан иначе

Если автоматический поиск не нашёл маркер, пользователь вручную вводит строку и столбец. Автоматическое значение не должно перезаписывать подтверждённое ручное значение.

### 8.3. Другое число пар

Пользователь меняет `Pairs` в `tblParserProfiles`. Парсер обрабатывает заданное количество пар, а не жёстко четыре.

### 8.4. Другое число строк на день или пару

Параметры `RowsPerDay`, `RowsPerPair`, `CodeOffset`, `SubjectOffset`, `RoomOffset` меняются вручную.

### 8.5. Легенда отсутствует

Занятия могут быть извлечены, но преподаватели остаются неназначенными. В листе контроля появляется причина, а не случайный преподаватель.

### 8.6. Неверный профиль

Ошибочный источник не должен останавливать обработку всех остальных источников. Его статус и ошибка записываются отдельно.

## 9. Преподаватели и алиасы

### 9.1. Точное ФИО

Точное нормализованное совпадение получает преподавателя без диалога.

### 9.2. Фамилия и инициалы

Вариант `Иванов И.И.` сопоставляется по алиасу либо по однозначному преподавателю.

### 9.3. Два преподавателя с фамилией Иванов

Автоматическое назначение по одной фамилии запрещается. Пользователь должен выбрать TeacherID и сохранить алиас.

### 9.4. Неактивный преподаватель

Строка `Активен = false` исключается из новых автоматических назначений.

### 9.5. Преподаватель удалён после предыдущего расчёта

Прежнее назначение не сохраняется, если TeacherID больше не существует или неактивен. Занятие перераспределяется либо остаётся неназначенным.

## 10. Алгоритм назначения

Приоритет:

1. ручное закрепление;
2. прежнее допустимое назначение;
3. подтверждённый поток;
4. единственный кандидат;
5. Round-Robin;
6. неназначенное занятие или совместимый конфликт.

### 10.1. Round-Robin

Результат не зависит от случайного порядка файлов. Перед распределением занятия сортируются по устойчивым ключам.

### 10.2. Ручное закрепление

Запись в `tblManualOverrides` с режимом `lock` имеет приоритет над автоматическим распределением.

### 10.3. Ручное исключение

Режим `exclude` удаляет преподавателя из кандидатов конкретного занятия.

### 10.4. Сохранение прежнего назначения

При обновлении источника прежний TeacherID сохраняется, если он по-прежнему допустим и не создаёт конфликт.

## 11. Потоковые занятия

### 11.1. Допустимый поток

Объединение возможно при совпадении недели, дня, пары, дисциплины, вида занятия, аудитории и совместимого преподавателя.

### 11.2. Разные аудитории

Занятия не объединяются автоматически в один поток.

### 11.3. Лекция и практика

Разные виды занятий не считаются одним потоком даже при одинаковой дисциплине и времени.

### 11.4. Предполагаемый поток

Автоматически найденный поток должен быть виден пользователю и допускать ручное разделение.

## 12. Конфликты

### 12.1. Преподаватель занят

Строгий режим оставляет занятие неназначенным и создаёт критическую запись.

### 12.2. Совместимый режим

Допускается назначение с явным критическим конфликтом. Оно не должно выглядеть как корректное.

### 12.3. Одна группа в двух местах

Создаётся конфликт группы независимо от того, разные ли преподаватели.

### 12.4. Одна аудитория занята

Создаётся конфликт аудитории независимо от дисциплины.

### 12.5. Полный дубль занятия

Полный дубль отделяется от ресурсного конфликта и имеет собственный тип диагностики.

## 13. Обновление источников

### 13.1. Изменился один групповой лист

Переразбирается изменённый источник. Назначения пересчитываются глобально с сохранением допустимых прежних решений.

### 13.2. Источник удалён

Связанные занятия, кандидаты и назначения удаляются каскадно. Остальные источники сохраняются.

### 13.3. Изменилось название группы в выбранной ячейке

При режиме `cell` новое значение используется при следующем обновлении.

### 13.4. Источник временно повреждён

Предыдущий результат не должен молча считаться новым успешным расчётом. Источник получает статус ошибки.

## 14. Результирующие листы

### 14.1. Сводное расписание

Строится из нормализованных назначений, а не из визуальной геометрии исходных листов.

### 14.2. Выбранная неделя

Пользователь меняет номер недели и получает соответствующее представление без создания десятков постоянных листов.

### 14.3. Печатные недели

Физические листы создаются только по команде экспорта/печати.

### 14.4. Нагрузка преподавателей

Суммируются занятия по преподавателю и неделе. Неназначенные занятия не приписываются случайному сотруднику.

### 14.5. Занятость аудиторий

Показывается по нормализованным занятиям, включая выявленные конфликты.

## 15. Версии и восстановление

### 15.1. Перед пересчётом

Создаётся снимок назначений.

### 15.2. Восстановление

После восстановления повторно выполняется аудит конфликтов. Старый снимок не считается автоматически корректным для изменённых справочников.

### 15.3. Ручные закрепления

Хранятся отдельно от автоматически рассчитанных назначений и не исчезают при обычном пересчёте.

## 16. Установка и обновление

### 16.1. Per-user установка

Административные права не требуются. XLAM, XLTM и служебные файлы устанавливаются в профиль пользователя.

### 16.2. Excel открыт

Установка XLAM останавливается с понятным сообщением, чтобы не повредить загруженную надстройку.

### 16.3. Обновление существующей версии

Перед заменой создаётся резервная копия. При ошибке выполняется откат.

### 16.4. Только шаблон

Режим `TemplateOnly` не требует регистрации XLAM.

### 16.5. Только Office.js

Режим `ModernOnly` копирует манифесты и не требует Excel COM.

### 16.6. Удаление

Снимается регистрация XLAM, удаляется шаблон, ярлыки и Office.js payload. Журнал удаления сохраняется.

## 17. macOS

### 17.1. Browser preview

`npm start` работает без PowerShell и показывает функциональную страницу на localhost.

### 17.2. Sideload

`npm run sideload:mac` копирует манифест в папку `wef`. После первого запуска Excel перезапускается.

### 17.3. Legacy binary

На Mac можно редактировать исходники и собирать современный контур. Автоматическая компиляция XLAM/XLTM выполняется на Windows runner либо вручную в Excel.

## 18. Безопасность и данные

- исходные листы не отправляются на внешний backend;
- служебные данные хранятся в текущей книге;
- браузерный preview работает на демонстрационных аргументах;
- тестовые фикстуры не должны содержать реальные персональные данные;
- макросные пакеты должны быть подписаны перед корпоративным развёртыванием;
- удаление дублей и массовая замена требуют явного действия пользователя.

## 19. Критерий готовности релиза

Версия считается готовой, когда:

1. `npm ci` успешно выполняется на чистой машине;
2. `npm run check` проходит в Linux и Windows CI;
3. localhost dashboard не пустой;
4. browser preview работает без Excel;
5. Excel runtime включает операции книги;
6. `functions.json` генерируется автоматически;
7. оба манифеста проходят проверку;
8. Inno Setup проект компилируется;
9. XLAM/XLTM собираются и открываются на Windows runner с Excel;
10. выполнена ручная приёмка хотя бы на одном Microsoft 365 Excel и одном legacy Excel.
